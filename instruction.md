Our internal auth gateway at /app (a small Go service, endpoints documented in /app/docs/API.md) got flagged in a security review: someone without admin credentials can end up with a working session against /admin/users. There's more than one hole causing this, not just a single bug, so don't stop after finding the first one.

Concretely: there should be no way for someone without real admin credentials to end up holding a session or token the server accepts as admin — whether that's something they send directly, or a credential they're able to construct or recover on their own. When you're auditing this, be thorough — a review that only skims the .go files and calls it done isn't complete, and don't assume you're done just because you've tightened how a token's format or algorithm is checked; think about whether the material backing it could itself be compromised. Fix it directly so every path in is closed, but don't break anything that's supposed to work — registration, normal password login, the remember-me cookie flow, the admin-only service-token endpoint, and the audit log all need to keep behaving exactly as documented. The RSA keypair, the /.well-known/jwks.json endpoint, and the /healthz fields exist for real features, so don't just delete them as a shortcut to "fixing" this.
