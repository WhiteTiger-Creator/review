The mesh security team needs an offline gate before reenabling workload-to-workload calls on a production cluster. Build a Go reviewer that evaluates the signed SPIFFE call bundle and reports which requests are allowed or denied. The bundle includes rotating workload and approver keys, transitive scope implications, overlapping network zones, delegation chains, approval quorums, and replay state.

Create `/app/workload_gate.go` as a standalone Go `package main`. The harness builds it with `/usr/local/go/bin/go build -o /tmp/workload-gate /app/workload_gate.go` and runs `/tmp/workload-gate <input_dir> <output_dir>`, using `/app/task_file/input_data` and `/app/task_file/output_data` for the public bundle. The program must create the output directory if needed, overwrite `workload_decisions.jsonl` and `workload_summary.json`, and leave unrelated files alone.

Use `/app/task_file/docs/workload_gate_contract.md` as the authoritative schema and policy contract. It defines the HMAC strings, key validity rules, base64url handling, scope implication closure, route matching, identity checks, delegation chains, approval quorum rules, nonce replay state, reason order, output schema, null behavior, and sorting. Exact-match grading is used after JSON parsing for field names, lowercase enum strings, nulls, row order, reason order, scope order, delegation ID order, approval ID order, summary fields, and counts.

Do not modify anything under `/app/task_file/input_data` or `/app/task_file/docs`. The grader rebuilds the same Go source and reruns it on compatible changed security bundles, so a checked-in report is not enough. The Go program must do the review in-process and must not shell out, invoke Python or pytest, read `/tests`, or reference verifier files or reward files.
