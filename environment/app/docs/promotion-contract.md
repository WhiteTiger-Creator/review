# Promotion contract

The input files are JSON. `registry/profiles.json` is the profile catalog. `requests/requests.json` is an ordered list. `policies/stages.json` contains stage policies. The default authority URL is `https://huggingface.co/api/models/google-bert/bert-base-uncased`; `REGISTRY_URL`, when present, replaces that full URL. `/app/solve.sh` is the rerunnable entrypoint and must perform a fresh GET to the selected URL for every request, including requests later rejected. A profile is registry-live only when its `authority_model` equals the nonempty `id` returned by the selected authority. A failed request or a different id is a registry failure and must not produce a promotion.

A request names a profile, target stage, task family, and requested inference settings. Resolve a profile by exact profile id. A profile is registry-live only if its `authority_model` equals the nonempty `id` returned by the authority. A profile is eligible only if it is registry-live, `status` is `active`, `retired` is false, and its `family` equals the request family. The profile's `stage_hint` is descriptive and never decides eligibility.

Stage policies are selected by exact stage. Their limits are `min_context`, `max_batch`, and `allowed_quantization`. The effective context is the profile context limit reduced by the request's `context_reserve`. The effective batch is the smaller of the profile batch limit and request batch. The request is compatible when effective context is at least the policy minimum, effective batch is at most the policy maximum, and the profile quantization is in the allowed list. Context reserve is applied before the stage comparison. A zero reserve is valid.

Decision precedence is `profile-missing` (the requested profile is absent), `registry-missing` (the profile exists but is not registry-live), `inactive-profile`, `family-mismatch`, `stage-unknown`, `context-incompatible`, `batch-incompatible`, `quantization-incompatible`, `promoted`. The first applicable reason is used. A promoted request resolves to its requested stage. A rejected request resolves to `none` and has no selected profile, represented by an empty `profile_id`.

Each decision has keys in this order: `request_id`, `profile_id`, `stage`, `effective_context`, `effective_batch`, `quantization`, `decision`, `reason`. Requests rejected before stage-policy comparison, namely `profile-missing`, `registry-missing`, `inactive-profile`, `family-mismatch`, and `stage-unknown`, have zero numeric fields and an empty quantization. For a request that reaches stage-policy comparison, compute the effective context and batch before applying the three compatibility checks; those computed numeric values remain in a rejected decision, while its quantization remains empty. The top-level object has keys `schema`, `decisions`, `source`, `digest`. `schema` is `model-promotion/v3`, and `source` is the authority response `id`. Digest is the lowercase SHA256 of the UTF-8 bytes of the compact JSON encoding of an object containing `schema`, `decisions`, and `source`, with those keys in that order. The digest field is excluded from the bytes being hashed.

The request with a context reserve equal to its profile context limit demonstrates that reserve changes eligibility. The request whose profile advertises a different stage hint demonstrates that the hint is not authority. A profile with batch capacity above the request still uses the smaller effective batch. These are ordinary consequences of the rules and are part of the required plan.
