The reviewer reads four files from the input directory: `mesh_policy.json`, `workloads.json`, `approvers.json`, and `calls.jsonl`. All timestamps are UTC RFC3339 strings with a trailing `Z`. Base64url values are unpadded. Arrays in the input may be in any order except `calls.jsonl`, which is evaluated in file order.

`mesh_policy.json` has `as_of`, `trusted_domains`, `allowed_audiences`, `network_zones`, `default_network_risk`, `scope_implications`, and `routes`. A network zone has `cidr` and integer `risk`; use the matching zone with the most-specific CIDR prefix whose CIDR contains `origin_ip`, breaking exact prefix-length ties by earlier array order. If `origin_ip` is not a valid IP address or no zone matches, use `default_network_risk`. `scope_implications` maps a scope string to scopes it implies. Scope normalization removes blanks and duplicates, then computes the transitive implication closure, including the original scopes, and finally sorts lexicographically. The closure must terminate even when implications contain cycles. A route object has `method`, `path_prefix`, `allowed_subject_groups`, `allowed_source_clusters`, `allowed_envs`, `required_scopes`, `max_network_risk`, `max_assertion_age_seconds`, `require_attestation`, `allow_delegation`, `max_delegation_hops`, and `required_approval_roles`. Attestation order is `none < baseline < hardware`.

`workloads.json` has `workloads`, an array keyed by unique `spiffe_id`. A workload has `spiffe_id`, `service`, `namespace`, `trust_domain`, `cluster`, `env`, `groups`, `status`, `attestation`, and `keys`. Its identity fields match only when `spiffe_id` is exactly `spiffe://<trust_domain>/ns/<namespace>/sa/<service>`. `approvers.json` is an array keyed by unique `approver_id`; each approver has `approver_id`, `roles`, `status`, and `keys`. A key has `key_id`, `secret_b64url`, `status`, `not_before`, and `expires_at`. A key is usable only when its status is exactly `active` and `not_before <= validation_time < expires_at`.

Each `calls.jsonl` row has `request_id`, `route_id`, `source_id`, `subject_id`, `key_id`, `method`, `path`, `audience`, `nonce`, `origin_ip`, `assertion_b64url`, `signature_b64url`, `delegations`, and `approvals`. Decode `assertion_b64url` as a JSON object with exactly the security claims `request_id`, `source_id`, `subject_id`, `route_id`, `key_id`, `method`, `path`, `audience`, `nonce`, `issued_at`, and `scopes`. The assertion is malformed if it is not base64url JSON, any listed string field is missing or empty, `issued_at` is not a timestamp, or `scopes` is not an array of strings. Assertion scopes are normalized with the policy's transitive scope implication closure.

The call signature is HMAC-SHA256 using the source workload key named by the call row's `key_id`. The key validation time is the decoded assertion `issued_at`. Sign this exact string:

`request_id + "\n" + source_id + "\n" + key_id + "\n" + assertion_b64url + "\n" + nonce`

The signature is checked only when the source workload is known and the assertion decoded correctly. A decoded assertion whose listed claims do not exactly equal the call row fields emits `assertion-claim-mismatch`; this does not stop other checks. `audience-not-allowed` uses the call row `audience`. `assertion-expired` is route-specific: when the route is known, emit it if `issued_at` is after `as_of` or if `as_of - issued_at` is greater than `max_assertion_age_seconds`.

Delegation is required when `source_id` and `subject_id` differ. A delegated call to a route with `allow_delegation: false` emits `delegation-not-allowed`; the chain is still checked and may also emit `delegation-invalid`. Direct calls must have an empty `delegations` array or they emit `delegation-invalid`. For delegated calls, the chain is valid only when all of these hold: the number of hops is at least 1 and no more than the known route's `max_delegation_hops`; the first hop `from` equals `subject_id`; every next hop starts at the previous hop's `to`; the final hop `to` equals `source_id`; every hop endpoint is a known active workload in a trusted domain; `not_before <= as_of < expires_at` for every hop; every hop has at least one scope; and every hop signature is valid.

A delegation has `grant_id`, `from`, `to`, `key_id`, `scopes`, `not_before`, `expires_at`, and `signature_b64url`. A delegation signature is HMAC-SHA256 using the `from` workload key named by the grant's `key_id`. The key validation time is the grant `not_before`. Sign this exact string:

`grant_id + "\n" + request_id + "\n" + from + "\n" + to + "\n" + key_id + "\n" + canonical_scopes + "\n" + not_before + "\n" + expires_at`

`canonical_scopes` is the hop's listed scope set sorted lexicographically and joined with commas before implication expansion. A valid delegated chain can still be too narrow: emit `delegation-scope-missing` when the route's required scopes are not all present in the intersection of every hop's expanded scope closure. `scope-missing` is checked separately against the expanded assertion scope set for every known route. For output, `accepted_delegation_ids` is the chain's `grant_id` values in input order only when the chain is valid and the known route allows delegation; otherwise it is `[]`.

Approvals are additive and are checked for every request. An approval is valid only when the approver exists, has `status: "active"`, the approval `decision` is exactly `approve`, the approval `role` is one of the approver's roles, `issued_at <= as_of < expires_at`, and the signature is valid. The approval signature is HMAC-SHA256 using the approver key over:

`approval_id + "\n" + request_id + "\n" + approver_id + "\n" + key_id + "\n" + role + "\n" + decision + "\n" + issued_at + "\n" + expires_at + "\n" + ticket`

The approver key is named by the approval's `key_id`, and the key validation time is the approval `issued_at`. If any approval on the request is invalid, emit `approval-invalid` once. For a known route, emit `approval-missing` unless each `required_approval_roles` entry can be covered by a valid approval with that role and the covering approvals use distinct `approver_id` values. Repeated role entries require repeated distinct approvers. Extra valid approvals may appear in `valid_approval_ids`; invalid approvals never appear there. `valid_approval_ids` is sorted lexicographically.

Nonce replay is stateful over the input file. After evaluating a row, remember the tuple `(source_id, audience, nonce)` from the call row even if that row is denied or malformed. Emit `nonce-replay` on a row when the same tuple appeared earlier. Replay is additive and is always evaluated after the other reason checks.

Emit each reason at most once per request, in this exact order when applicable:

`unknown-source`, `unknown-subject`, `unknown-route`, `source-blocked`, `subject-blocked`, `untrusted-domain`, `identity-mismatch`, `route-mismatch`, `subject-not-allowed`, `cluster-not-allowed`, `env-not-allowed`, `attestation-too-low`, `scope-missing`, `assertion-malformed`, `assertion-claim-mismatch`, `assertion-expired`, `audience-not-allowed`, `bad-signature`, `delegation-not-allowed`, `delegation-invalid`, `delegation-scope-missing`, `network-too-risky`, `approval-invalid`, `approval-missing`, `nonce-replay`

Route checks are additive. `route-mismatch` means method differs from the route method or path does not start with `path_prefix`. `subject-not-allowed` is checked only when the route and subject workload are known. `cluster-not-allowed`, `env-not-allowed`, and `attestation-too-low` are checked only when the route and source workload are known. `untrusted-domain` and `identity-mismatch` are each emitted once if either the known source or the known subject violates that rule.

`effective_scopes` is the expanded sorted assertion scope set for direct calls. For delegated calls with a valid chain, it is the sorted intersection of expanded assertion scopes and every expanded hop scope set. For delegated calls without a valid chain, it is `[]`. A malformed assertion always gives `effective_scopes: []`.

Write `workload_decisions.jsonl` with one JSON object per call in the same order as `calls.jsonl`. Each row has exactly these keys: `request_id`, `decision`, `reasons`, `route_id`, `source_id`, `subject_id`, `network_risk`, `effective_scopes`, `accepted_delegation_ids`, `valid_approval_ids`, and `authorization_expires_at`. `decision` is `allow` only when `reasons` is empty, otherwise `deny`. For denied rows, `authorization_expires_at` is `null`. For allowed rows, it is the earliest of the assertion freshness expiry, accepted delegation `expires_at` values, and every valid approval expiry whose role is required by the route. Format it as UTC RFC3339 with `Z`. All list-valued output fields must serialize as JSON arrays even when empty, never as `null`; in Go, initialize empty slices before encoding so empty values appear as `[]`.

Write `workload_summary.json` with exactly `total_requests`, `allowed`, `denied`, `allowed_request_ids`, `denied_request_ids`, and `reason_counts`. Request ID arrays preserve call order. `reason_counts` has every reason string listed above as a key, including zero counts.
